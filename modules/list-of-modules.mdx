BBOT ships with a large library of modules covering subdomain enumeration, web scanning, OSINT, cloud enumeration, output formats, and more. You can view the full list at any time from the command line:

```bash
bbot -l
```

## Understanding Flags

Flags are labels attached to modules that let you enable entire categories at once with `-f`. For example:

```bash
# Enable all subdomain enumeration modules
bbot -f subdomain-enum -t evilcorp.com

# Enable all passive, safe modules
bbot -f passive safe -t evilcorp.com
```

Common flags include:

| Flag | Meaning |
|------|---------|
| `active` | Makes real requests to targets |
| `passive` | Gathers data without touching the target directly |
| `safe` | Non-destructive; unlikely to cause harm |
| `aggressive` | May trigger security alerts or cause side-effects |
| `deadly` | Can cause real damage; requires `--allow-deadly` |
| `subdomain-enum` | Discovers subdomains |
| `web-basic` | Basic web scanning |
| `web-thorough` | More thorough web scanning |
| `cloud-enum` | Cloud resource enumeration |
| `code-enum` | Code repository enumeration |
| `email-enum` | Email address discovery |
| `social-enum` | Social media enumeration |
| `slow` | Takes significantly longer than typical modules |

## Internal Modules

Internal modules run automatically on every scan. You do not need to enable them explicitly, but you can disable them individually if needed.

<Note>
Disabling internal modules can severely impact scan quality. Only disable them if you know what you're doing.
</Note>

To disable an internal module, set it to `False` in your config:

```yaml
speculate: false
excavate: false
aggregate: false
dnsresolve: false
cloudcheck: false
```

| Module | Description |
|--------|-------------|
| `aggregate` | Summarizes scan statistics in a table at the end of every scan |
| `cloudcheck` | Tags events by cloud provider and identifies cloud resources like storage buckets |
| `dnsresolve` | Performs DNS resolution and wildcard detection |
| `excavate` | Passively extracts URLs, parameters, emails, errors, and other data from HTTP responses using YARA rules. Also applies [custom YARA rules](/modules/custom-yara-rules). |
| `speculate` | Infers event types from existing data — e.g. open ports from URLs, parent domains from DNS names |
| `unarchive` | Extracts compressed archives found on the filesystem during a scan |

### excavate

The `excavate` module is one of BBOT's most powerful internal modules. It passively searches every HTTP response for:

- **URLs** — feeds the recursive web spider
- **Web parameters** — GET/POST params, form fields, jQuery parameters (produces `WEB_PARAMETER` events)
- **Email addresses** — extracted from page content
- **Verbose errors** — detects error messages that reveal framework details
- **CSP domains** — extracts domains from `Content-Security-Policy` headers
- **Serialized objects** — detects Java, .NET, and PHP serialization
- **File upload fields and WSDL URLs**
- **Non-HTTP scheme URLs** — `ftp://`, `mailto:`, `javascript:`, etc.
- **Custom YARA rule matches** — any rules you supply via `--custom-yara-rules`

### speculate

The `speculate` module bridges gaps between event types so that scans can discover web resources even without a port scanner enabled:

- `IP_RANGE` → individual `IP_ADDRESS` events
- `DNS_NAME` → parent domain `DNS_NAME` events
- `URL` / `URL_UNVERIFIED` → `OPEN_TCP_PORT` events (when no port scanner is active)
- `SOCIAL` / `AZURE_TENANT` → `ORG_STUB` events
- `USERNAME` → `EMAIL_ADDRESS` events (when username looks like an email)
