BBOT scans are launched from the command line with `bbot`. At minimum, you provide a target with `-t`. From there you can add modules (`-m`), flags (`-f`), and presets (`-p`) to shape what the scan does.

## Scan names and output

Every BBOT scan gets a random, mildly-entertaining name like **`demonic_jimmy`**. Output for that scan — including scan stats and any web screenshots — is saved to a folder by that name in `~/.bbot/scans`. The most recent 20 scans are kept; older ones are removed automatically.

You can override the name with `-n` and the output directory with `-o`:

```bash
# save everything to the folder "my_scan" in the current directory
bbot -t evilcorp.com -f subdomain-enum -m gowitness -n my_scan -o .
```

If you reuse a scan name, BBOT appends to the previous output files rather than overwriting them.

## Targets (`-t`)

Targets declare what is in scope and seed the scan with initial data. BBOT accepts an unlimited number of targets on the command line, from a file, or both.

| Target type | Example |
|---|---|
| `DNS_NAME` | `evilcorp.com` |
| `IP_ADDRESS` | `1.2.3.4` |
| `IP_RANGE` | `1.2.3.0/24` |
| `OPEN_TCP_PORT` | `192.168.0.1:80` |
| `URL` | `https://www.evilcorp.com` |
| `EMAIL_ADDRESS` | `bob@evilcorp.com` |
| `ORG_STUB` | `ORG:evilcorp` |
| `USER_STUB` | `USER:bobsmith` |
| `FILESYSTEM` | `FILESYSTEM:/tmp/asdf` |
| `MOBILE_APP` | `MOBILE_APP:https://play.google.com/store/apps/details?id=com.evilcorp.app` |

<Note>
BBOT discriminates only down to the host level. If you specify `https://www.evilcorp.com` as a target, the scan is **seeded** with that URL, but the scope covers the entire host `www.evilcorp.com`. Other ports and URLs on that same host may also be scanned.
</Note>

You can load targets from a file and the command line simultaneously:

```bash
# targets.txt
4.3.2.1
10.0.0.2:80
1.2.3.0/24
evilcorp.com
evilcorp.co.uk
https://www.evilcorp.co.uk
```

```bash
# load targets from a file and from the command line
bbot -t targets.txt fsociety.com 5.6.7.0/24 -m portscan
```

## Modules (`-m`)

Modules are the engine of BBOT. They consume events (`IP_ADDRESS`, `DNS_NAME`, etc.) from each other, process the data, and emit new events. Enable individual modules with `-m`:

```bash
# enable modules: portscan, sslcert, and httpx
bbot -t www.evilcorp.com -m portscan sslcert httpx
```

To list all available modules:

```bash
bbot -l
```

### Types of modules

Modules fall into three categories:

<AccordionGroup>
  <Accordion title="Scan modules">
    The majority of modules. Examples: `portscan`, `sslcert`, `httpx`, `nuclei`, `certspotter`. Enable with `-m`.
  </Accordion>
  <Accordion title="Output modules">
    Output scan data to different formats and destinations. `txt`, `json`, and `csv` are enabled by default. Enable additional ones with `-om`. See [Output](/scanning/output) for the full list.
  </Accordion>
  <Accordion title="Internal modules">
    Always enabled. Perform essential background tasks:

    - **`aggregate`** — summarizes results at the end of a scan
    - **`excavate`** — extracts subdomains, emails, and other data from web pages
    - **`speculate`** — infers new events, e.g. `OPEN_TCP_PORT` from `URL`, or `IP_ADDRESS` from `IP_NETWORK`

    Disable them individually via config: `-c speculate=false`
  </Accordion>
</AccordionGroup>

### Module dependencies

BBOT installs module dependencies at runtime using Ansible. Dependency behavior can be controlled with these flags:

| Flag | Description |
|---|---|
| `--no-deps` | Don't install module dependencies |
| `--force-deps` | Force reinstall all module dependencies |
| `--retry-deps` | Retry failed module dependency installs |
| `--ignore-failed-deps` | Run modules even if dependencies failed |
| `--install-all-deps` | Pre-install dependencies for every module |

## Flags (`-f`)

Flags are how BBOT groups its modules. Using a flag enables every module that carries that flag — no need to list them individually.

```bash
# enable all subdomain enumeration modules
bbot -t evilcorp.com -f subdomain-enum

# list all modules that have the subdomain-enum flag
bbot -f subdomain-enum -l
```

### Filtering with flags

| Option | Description | Example |
|---|---|---|
| `-f` | Enable modules with these flags | `-f subdomain-enum` |
| `-rf` | Require modules to have this flag | `-rf passive` |
| `-ef` | Exclude modules with these flags | `-ef slow` |
| `-em` | Exclude specific modules | `-em ipneighbor` |
| `-lf` | List all available flags | `-lf` |

Every module is tagged as either `safe` or `aggressive`, and either `active` or `passive`. These are useful for filtering:

```bash
# enable safe modules but exclude active ones (equivalent to requiring passive)
bbot -t evilcorp.com -f safe -ef active

# same result using -rf
bbot -t evilcorp.com -f safe -rf passive
```

### Available flags

| Flag | Modules | Description |
|---|---|---|
| `safe` | 98 | Non-intrusive, safe to run |
| `passive` | 70 | Never connects to target systems |
| `active` | 52 | Makes active connections to target systems |
| `subdomain-enum` | 51 | Enumerates subdomains |
| `aggressive` | 22 | Generates a large amount of network traffic |
| `code-enum` | 18 | Find public code repositories and scan for secrets |
| `web-basic` | 17 | Basic, non-intrusive web scan |
| `cloud-enum` | 16 | Enumerates cloud resources |
| `web-thorough` | 15 | More advanced web scanning |
| `slow` | 11 | May take a long time to complete |
| `email-enum` | 9 | Enumerates email addresses |
| `affiliates` | 8 | Discovers affiliated hostnames/domains |
| `download` | 7 | Downloads files, apps, or repositories |
| `deadly` | 6 | Highly aggressive — requires `--allow-deadly` |
| `baddns` | 3 | Runs all BadDNS DNS audit modules |
| `web-paramminer` | 3 | Discovers HTTP parameters via brute-force |
| `iis-shortnames` | 2 | Scans for IIS Shortname vulnerability |
| `portscan` | 2 | Discovers open ports |
| `social-enum` | 2 | Enumerates social media |
| `service-enum` | 1 | Identifies protocols on open ports |
| `subdomain-hijack` | 1 | Detects hijackable subdomains |
| `web-screenshots` | 1 | Takes screenshots of web pages |

<Warning>
Modules tagged `deadly` (such as `nuclei`, `ffuf`, `legba`, `medusa`, `vhost`, and `lightfuzz`) are disabled by default. You must pass `--allow-deadly` to use them.
</Warning>

## Presets (`-p`)

Presets are YAML files that bundle targets, modules, flags, and config options into a reusable package. BBOT ships with a collection of built-in presets for common tasks:

```bash
# list available presets
bbot -lp

# run subdomain enumeration
bbot -t evilcorp.com -p subdomain-enum

# combine multiple presets
bbot -t evilcorp.com -p subdomain-enum web-basic

# use a custom preset file
bbot -p ./my_preset.yml
```

See [Presets](/scanning/presets) for a full list of built-in presets and instructions on writing your own.

## Scope

For pentesters and bug bounty hunters, staying in scope matters. Active modules like `nuclei` only touch in-scope resources.

By default, scope is whatever you specify with `-t`, including child subdomains. If you target `evilcorp.com`, then `www.evilcorp.com`, `mail.evilcorp.com`, etc. are also in scope.

### Scope distance

Because BBOT is recursive, every event it discovers is assigned a **scope distance** representing how many hops it is from your declared targets.

- `evilcorp.com` → scope distance `0` (in-scope)
- `1.2.3.4` (resolved from `www.evilcorp.com`) → scope distance `1`
- `ecorp.blob.core.windows.net` (PTR of `1.2.3.4`) → scope distance `2`

Most active modules only consume events at distance `0`. Passive modules like `asn` accept up to distance `1`. DNS resolution happens out to distance `2` by default.

By default BBOT only displays in-scope events. To see further:

```bash
# display out-of-scope events up to one hop away
bbot -t evilcorp.com -f subdomain-enum -c scope.report_distance=1
```

### Strict scope

To scan **only** the exact target hostname and none of its children:

```bash
bbot -t evilcorp.com -f subdomain-enum --strict-scope
```

<Note>
`--strict-scope` applies to targets and whitelists, but **not** blacklists. A blacklisted domain always blocks its children regardless.
</Note>

### Whitelists

Override what counts as in-scope. For example, seed the scan from `evilcorp.com` but restrict active modules to a specific IP range:

```bash
# seed with evilcorp.com, but restrict scope to 1.2.3.0/24
bbot -t evilcorp.com --whitelist 1.2.3.0/24 -f subdomain-enum -m portscan nuclei --allow-deadly
```

### Blacklists

Blacklists take ultimate precedence. Anything blacklisted is completely excluded, even if it's in the whitelist:

```bash
# scan evilcorp.com, but exclude internal.evilcorp.com and its children
bbot -t evilcorp.com --blacklist internal.evilcorp.com -f subdomain-enum -m portscan nuclei --allow-deadly
```

Blacklists also accept regex patterns prefixed with `RE:`. These are checked against the full URL including host and path:

```bash
# exclude all URLs containing "signout"
bbot -t evilcorp.com --blacklist "RE:signout"

# narrow the pattern to only match the URL path
bbot -t evilcorp.com --blacklist 'RE:signout\.aspx$'
```

## DNS wildcards

BBOT has built-in wildcard detection. It tags wildcard domains and collapses wildcard hosts under a `_wildcard` prefix:

```text
[DNS_NAME]  _wildcard.github.io  TARGET  (a-record, a-wildcard, a-wildcard-domain, ...)
```

To disable wildcard detection for specific domains:

```yaml title="~/.config/bbot/bbot.yml"
dns:
  wildcard_ignore:
    - evilcorp.com
    - evilcorp.co.uk
```

If wildcard detection is unreliable in edge cases (e.g., dynamic DNS), increase the number of sanity checks:

```yaml title="~/.config/bbot/bbot.yml"
# default is 10
dns:
  wildcard_tests: 20
```
