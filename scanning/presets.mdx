title: "Presets"
description: "Bundle scan settings into reusable YAML files. Use built-in presets or create your own with custom modules, flags, and config options."
---

Once you start customizing BBOT, your commands can get long. Presets let you put all your scan settings in a single YAML file and reference it with `-p`:

```bash
bbot -p ./my_preset.yml
```

A preset can include targets, modules, flags, output modules, blacklists, and config options like API keys.

---

## Using presets

BBOT ships with a ready-made collection of presets for common tasks. They live in `~/.bbot/presets`.

```bash
# list available presets
bbot -lp

# run subdomain enumeration
bbot -t evilcorp.com -p subdomain-enum

# combine multiple presets
bbot -t evilcorp.com -p subdomain-enum spider

# start with a preset but only enable passive modules
bbot -t evilcorp.com -p subdomain-enum -rf passive

# preset with a config override
bbot -t www.evilcorp.com -p spider -c web.spider_distance=10
```

## Writing a custom preset

A preset is a YAML file. Here is a minimal example — the built-in `subdomain-enum` preset:

```yaml title="subdomain-enum.yml"
description: Enumerate subdomains via APIs, brute-force

flags:
  - subdomain-enum

output_modules:
  - subdomains
```

Here is a more complete custom preset that builds on two built-in presets:

```yaml title="my_subdomains.yml"
description: Subdomain enumeration + basic web scan + nuclei

target:
  - evilcorp.com

include:
  # include these built-in presets
  - subdomain-enum
  - web-basic

modules:
  # enable nuclei in addition to the other modules
  - nuclei

config:
  # global config options
  web:
    http_proxy: http://127.0.0.1:8080
  # module config options
  modules:
    securitytrails:
      api_key: 21a270d5f59c9b05813a72bb41707266
    virustotal:
      # multiple API keys are allowed
      api_key:
        - 4f41243847da693a4f356c0486114bc6
        - 5bc6ed268ab6488270e496d3183a1a27
```

```bash
bbot -p ./my_subdomains.yml
```

### Preset fields

| Field | Description |
|---|---|
| `description` | Human-readable description of what the preset does |
| `target` | List of scan targets (same types as `-t`) |
| `include` | Other presets to inherit from |
| `modules` | Scan modules to enable |
| `output_modules` | Output modules to enable |
| `flags` | Enable all modules with these flags |
| `exclude_modules` | Modules to disable |
| `blacklist` | Targets to exclude from the scan |
| `config` | Config options (API keys, module settings, etc.) |
| `conditions` | Custom Jinja conditions evaluated before the scan starts |
| `module_dirs` | Additional directories to load custom modules from |

## Preset inheritance

Use the `include` key to build on existing presets:

```yaml title="my_thorough_scan.yml"
description: Full enumeration with screenshots

include:
  - subdomain-enum
  - web-basic
  - web-screenshots
```

### Load order matters

When you enable multiple presets, the **last one wins** in case of a conflict. For example, if `my_spider.yml` sets `web.spider_distance` to 1:

```yaml title="my_spider.yml"
config:
  web:
    spider_distance: 1
```

Then running:

```bash
# spider preset wins — overrides my_spider.yml
bbot -t evilcorp.com -p ./my_spider.yml spider
```

...means `spider` (loaded last) overrides `my_spider.yml`. Reverse the order to keep your value:

```bash
# my_spider.yml wins — loaded last
bbot -t evilcorp.com -p spider ./my_spider.yml
```

## Validating a preset

To confirm BBOT is configured the way you expect before running a scan:

```bash
# show the final merged preset that would be used
bbot -p ./my_preset.yml --current-preset

# show the full preset including all defaults
bbot -p ./my_preset.yml --current-preset-full
```

## Advanced preset features

### Custom modules

Load a custom `.py` module from a local directory:

```yaml title="custom_modules.yml"
# load extra BBOT modules from this location
module_dirs:
  - /home/user/custom_modules
```

### Environment variable substitution

Reference environment variables in your preset with `${env:<variable>}`:

```yaml title="my_nuclei.yml"
description: Nuclei scan with runtime template selection

target:
  - evilcorp.com

modules:
  - nuclei

config:
  modules:
    nuclei:
      tags: ${env:NUCLEI_TAGS}
```

```bash
NUCLEI_TAGS=apache,nginx bbot -p ./my_nuclei.yml
```

### Conditions

Add custom pre-scan logic with `conditions`. These use [Jinja](https://palletsprojects.com/p/jinja/) and run in a sandboxed environment:

```yaml title="abort_if_missing.yml"
description: Abort if nuclei templates aren't set

modules:
  - nuclei

conditions:
  - |
    {% if not config.modules.nuclei.templates %}
      {{ abort("Don't forget to set your templates!") }}
    {% endif %}
```

```yaml title="disable_ffuf_with_spider.yml"
description: Warn and disable ffuf when the spider is enabled

modules:
  - ffuf

conditions:
  - |
    {% if config.web.spider_distance > 0 and config.web.spider_depth > 0 %}
      {{ warn("Disabling ffuf because the web spider is enabled") }}
      {{ preset.exclude_module("ffuf") }}
    {% endif %}
```

Available variables in conditions:

| Variable | Description |
|---|---|
| `preset` | The current preset object |
| `config` | Alias for `preset.config` |
| `warn(message)` | Display a warning to the user |
| `abort(message)` | Abort the scan with an optional message |

### Web-specific presets

| Preset | Description | Modules |
|---|---|---|
| `dirbust-light` | Basic directory brute-force (surface level only) | 4 |
| `dirbust-heavy` | Recursive directory brute-force (aggressive) | 5 |
| `paramminer` | Discover web parameters via brute-force | 6 |
| `iis-shortnames` | Recursively enumerate IIS shortnames | 3 |
| `dotnet-audit` | Comprehensive IIS/.NET module scan | 9 |
| `web-screenshots` | Take screenshots of web pages via gowitness | 3 |
| `tech-detect` | Detect technologies via Nuclei and FingerprintX | 3 |
| `baddns-intense` | Run all BadDNS modules and submodules | 4 |

### Lightfuzz presets

| Preset | Description |
|---|---|
| `lightfuzz-light` | Path, SQLi, XSS only; no POST requests; safest to combine with larger scans |
| `lightfuzz-medium` | All lightfuzz modules; no POST requests (recommended starting point) |
| `lightfuzz-heavy` | All modules including POST; adds paramminer for parameter discovery |
| `lightfuzz-superheavy` | Most intense; tests each unique parameter-value individually |
| `lightfuzz-xss` | GET-based XSS only |

### Nuclei presets

| Preset | Description |
|---|---|
| `nuclei` | Run nuclei against all discovered targets (directory-only mode) |
| `nuclei-budget` | Nuclei in budget mode — low request count, good for quick wins |
| `nuclei-intense` | Nuclei against ALL URLs; best combined with `spider` |
| `nuclei-technology` | Nuclei templates matched to detected technologies |

    ```yaml title="~/.bbot/presets/spider.yml"
        # maximum number of links to follow per page
        spider_links_per_page: 25
    ```
